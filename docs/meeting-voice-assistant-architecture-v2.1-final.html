0 Review kiến trúc đề xuất

Dưới đây là đánh giá từng thành phần anh đề xuất, kèm điều chỉnh.

Điều chỉnh — FastAPI / Node gateway

Anh đề xuất dùng cả FastAPI lẫn Node gateway. Hai layer gateway sẽ thêm 1-2ms latency mỗi hop và tăng complexity. Đề xuất: Dùng FastAPI làm duy nhất gateway — FastAPI đã hỗ trợ native WebSocket, async/await, và truy cập trực tiếp CTranslate2 Python API. Node.js không cần thiết trừ khi scale >100 concurrent WebSocket connections. Với use case họp nội bộ (5-20 người), FastAPI single process + uvicorn workers đủ sức.

Giữ nguyên — NLLB + CTranslate2

Lựa chọn xuất sắc. CTranslate2 tối ưu đặc biệt cho NLLB — int8 quantization giảm 2-4x memory, inference ~30-80ms/câu trên GPU. NLLB-200-distilled-1.3B là sweet spot cho JA/KO/EN→VI: nhỏ hơn 3.3B (nhanh hơn 2x) nhưng chất lượng tốt hơn 600M đáng kể cho cặp ngôn ngữ thấp tài nguyên như KO→VI.

Giữ nguyên — WebSocket cho text delta

Đúng pattern. WebSocket persistent connection loại bỏ HTTP overhead cho mỗi message. Text delta (chỉ gửi phần thay đổi) giảm bandwidth và cho phép streaming UI — user thấy kết quả ngay khi có token đầu tiên.

Điều chỉnh — AI model qua 9router

9router/Router9 là cloud LLM gateway — route tới OpenAI/Anthropic/Google. Nếu mục tiêu là self-hosted hoàn toàn, anh nên chạy LLM local bằng vLLM (engine tốt nhất cho decoder-only models, hỗ trợ continuous batching + PagedAttention). Model đề xuất: Qwen2.5-7B-Instruct (hiểu tốt EN/JA/KO/VI, chạy trên 1x RTX 4090). Nếu anh vẫn muốn option gọi cloud LLM backup, dùng LiteLLM (self-hosted, open-source) thay cho 9router — nó cung cấp OpenAI-compatible API cho cả local model lẫn cloud providers.

Bổ sung — Thiếu component quan trọng

1. Redis — cần cho conversation context store (sliding window 5 turns) và session management giữa WebSocket reconnects. 2. Language Detection — SenseVoice on-device đã tự detect EN/JA/KO, nhưng server cần biết ngôn ngữ nguồn để chọn đúng NLLB language code. Client phải gửi kèm detected_lang trong mỗi message. 3. Health check + Graceful degradation — Nếu LLM server quá tải, pipeline vẫn phải trả translation (skip AI suggest) thay vì block toàn bộ.

Tại sao single FastAPI gateway thay vì FastAPI + Node.js?

FastAPI + uvicorn (ASGI) xử lý WebSocket native, async I/O tương đương Node.js cho workload này. Dùng single gateway giúp: (1) giảm 1-2ms inter-service hop, (2) FastAPI truy cập trực tiếp CTranslate2 Python API — không cần serialize qua HTTP/gRPC tới separate ML service, (3) giảm infrastructure complexity. Nếu sau này cần scale >100 concurrent connections → tách thành Node.js WS gateway + FastAPI ML backend.

So sánh: Self-hosted vs Cloud API

Translation self-hosted (30-80ms) nhanh hơn 2-3x so với Google Cloud API (100-250ms) vì loại bỏ internet round-trip. Server đặt cùng LAN hoặc cùng building → latency network gần như zero. Tuy nhiên, AI suggest self-hosted (Qwen2.5-7B) có TTFT ~300-500ms, chậm hơn chút so với Gemini Flash cloud (~250ms) nhưng trade-off chấp nhận được khi đổi lại toàn quyền kiểm soát data.

3.2 Translation Worker

ComponentChoiceLý do
Model NLLB-200-distilled-1.3B (int8) Khuyên dùng — 1.3B params sweet spot: chất lượng KO/JA→VI tốt hơn 600M rõ rệt, inference vẫn nhanh (~50ms/câu GPU)

Tokenizer SentencePiece (bundled) NLLB sử dụng SPM tokenizer. Load 1 lần khi start, reuse cho mọi request

# Translation Worker — khởi tạo 1 lần import ctranslate2, sentencepiece translator = ctranslate2.Translator( "models/nllb-1.3b-distilled-ct2", device="cuda", device_index=0, compute_type="int8", # int8 trên GPU: nhanh + ít VRAM inter_threads=1, intra_threads=4 ) sp = sentencepiece.SentencePieceProcessor() sp.load("models/nllb-1.3b-distilled-ct2/sentencepiece.bpe.model") # Hàm dịch — gọi từ FastAPI handler LANG_MAP = {"en":"eng_Latn", "ja":"jpn_Jpan", "ko":"kor_Hang"} def translate(text: str, src_lang: str) -> str: src_code = LANG_MAP[src_lang] tokens = sp.encode(text, out_type=str) tokens = [src_code] + tokens # NLLB cần prefix language code results = translator.translate_batch( [tokens], target_prefix=[["vie_Latn"]], beam_size=2, # beam=2 đủ tốt, nhanh hơn beam=4 max_input_length=512, max_decoding_length=256 ) output_tokens = results[0].hypotheses[0][1:] # bỏ lang prefix return sp.decode(output_tokens)

4.3 Server-side orchestration logic

# FastAPI WebSocket handler — orchestration core @app.websocket("/ws/{session_id}") async def ws_handler(ws: WebSocket, session_id: str): await ws.accept() context = await redis.get_context(session_id) or [] try: async for msg in ws.iter_json(): if msg["type"] == "stt_partial": # Speculative translation — debounced if should_translate_partial(msg): translation = await translate(msg["text"], msg["lang"]) await ws.send_json({ "type": "translation", "text": translation, "is_final": False }) elif msg["type"] == "stt_final": # Fire translation + AI suggest CONCURRENTLY async def do_translate(): t = await translate(msg["text"], msg["lang"]) await ws.send_json({"type":"translation", "text":t, "is_final":True}) async def do_suggest(): async for delta in get_suggestion(context, msg["text"], msg["lang"]): await ws.send_json({"type":"ai_suggest", "delta":delta, "done":False}) await ws.send_json({"type":"ai_suggest", "delta":"", "done":True}) # Concurrent — translation + AI cùng lúc await asyncio.gather(do_translate(), do_suggest()) # Update context in Redis context.append({"role":"user", "content":msg["text"]}) if len(context) > 10: context = context[-10:] await redis.set_context(session_id, context) except WebSocketDisconnect: pass

Tại sao không dùng LiveKit + SeamlessStreaming?

LiveKit — Skip cho v1

LiveKit là WebRTC platform cho multi-participant rooms. Use case hiện tại là personal assistant (1 phone), không phải conferencing. WebSocket đủ cho text deltas. LiveKit Agents giả định audio stream lên server — ta đã làm STT on-device nhanh hơn.

SeamlessStreaming — Xem xét v2

SeamlessStreaming (Meta) dịch đồng thời nhưng latency ~2s — chậm hơn 5x so với pipeline SenseVoice (300ms) + NLLB (50ms) = 350ms. Phải stream audio lên server (bandwidth + privacy). Speculative translation trong pipeline hiện tại đã giải quyết 80% nhu cầu dịch-khi-đang-nói.

Lưu ý GPU contention

NLLB và Qwen chạy cùng GPU sẽ tranh chấp CUDA streams. Khi LLM đang generate tokens (liên tục), translation request có thể bị queue 10-20ms. Mitigations: (1) CTranslate2 dùng separate CUDA stream, (2) Set vLLM --gpu-memory-utilization 0.7 để chừa headroom, (3) Nếu có budget → dùng 2 GPU (Option B).

Option B: Dual GPU (khuyên dùng cho production)

GPUServiceVRAM cần
GPU 0 (RTX 3090/4070)NLLB-1.3B int8 (CTranslate2)~2GB — dư rất nhiều
GPU 1 (RTX 4090)Qwen2.5-7B AWQ (vLLM)~14GB with KV cache

Dual GPU loại bỏ hoàn toàn contention. Translation luôn <50ms bất kể LLM load.
